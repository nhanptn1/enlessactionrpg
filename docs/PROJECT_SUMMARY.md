## 2. Skills & Lessons

Same base playbook as TopDownAdventure: the **`godot-2d-game-dev`** skill (`C:\Users\nhanp\.claude\skills\godot-2d-game-dev\SKILL.md`) covers the general techniques (verify loop, sprite extraction pipeline, Plan Mode conventions, correctness checklist, GH Pages deploy). What's specific to *this* project:

- **Strategy-pattern behaviors over full component-nodes.** `MovementBehavior`/`AttackBehavior` are stateless `Resource` subclasses shared across every `EnemyData` that references them (Godot dedupes `.tres` Resources by path at runtime â€” several species really do share one instance). This was a deliberate choice over splitting `EnemyBase` into full component Nodes (Health/Hitbox/HitFlash as separate nodes), made when comparing against an external architecture template â€” the reasoning: this project already commits to Resource-driven data everywhere, and full component-nodes would be inconsistent with that for no functional gain at this project's scale. **Any new behavior subclass must never store per-instance state on itself** â€” only read/write fields on the `enemy` node argument passed to every method.
- **The `:=` type-inference gotcha bit this project twice.** `var x := some_expr` fails to compile if the expression's static type resolves to `Variant` â€” most commonly when a variable typed as plain `Node` (e.g. from `get_first_node_in_group()`) has a member accessed that only exists on a subtype like `Node2D` (`.global_position`). Fix: use plain `=` instead of `:=` for anything chained off a generically-typed lookup, or explicitly type the intermediate variable (e.g. `Array[Node2D]` return types sidestep this entirely).
- **Don't assume a Godot API exists â€” verify it.** `GameManager` originally connected to `get_tree().current_scene_changed` to reset pause state on scene reload; that signal doesn't exist on `SceneTree` in Godot 4 at all, and it silently would have been a real runtime error the first time a Restart button was pressed. Fixed by adding an explicit `GameManager.reset_state()` method called directly from both restart handlers, right before `get_tree().reload_current_scene()` â€” more robust than a signal hook anyway, since it's deterministic rather than dependent on scene-tree timing.
- **`SceneTree.create_timer()` defaults `process_always` to `true`.** Any gameplay-critical timer (the boss's telegraph/cooldown/summon loop, a trap's arm/expire timers) needs `create_timer(duration, false)` explicitly, or it keeps counting down in real time even while `get_tree().paused = true` â€” the boss could keep landing hits on a "paused" screen otherwise. This is *not* what you want for music/UI-adjacent timers (e.g. the post-boss-victory music-restart delay), which should keep the default.
- **Centralize signal emission at the state-owner, not at every caller.** `SignalBus.enemy_hit` was originally emitted separately from `projectile.gd` and `trap_zone.gd` (each place that deals damage) â€” this silently missed the "instant_bomb" item effect, which damages enemies directly. Moved into `EnemyBase.take_damage()`/`BossBase.take_damage()` themselves so every current and future damage source is covered by construction.
- **No proactive headless test runs** â€” a standing, explicit user preference for this project specifically (headless Godot runs were found to hang/feel stuck). Verify changes via direct code re-reading instead, and say so plainly when a change hasn't been playtested in-editor.
- **Confirm before every `git push`**, every time, even mid-flow.
