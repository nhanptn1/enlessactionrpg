## 1. Summary

**What it is**: a Godot 4.7 vertical-portrait (720x1280) wave-survival auto-battler ‚Äî a fixed-position archer at the bottom of the screen, enemies fall from the top, auto-fire combat, XP/leveling, boss every 10 waves. Architecture is deliberately **per-Resource data-driven** (`EnemyData`/`SkillData`/`ItemData`/`WaveData`, plus `MovementBehavior`/`AttackBehavior` Strategy-pattern Resources) ‚Äî this is a real, intentional divergence from the sibling TopDownAdventure project's const-Dictionary pattern; the two are not meant to converge.

**Current playable scope**:
- **Player**: free to move left/right (A/D or mouse-drag, clamped to the play area). Two fully independent skill tracks, both auto-fired, both able to run at once: (1) the physical/basic line ‚Äî a single active skill at a time, upgraded wholesale (not stacked) via wave-clear player picks (Basic Shot ‚Üí Multishot ‚Üí Piercing Arrow ‚Üí Arrow Rain ‚Üí Trap Shot), pure physical damage, no status effects ‚Äî **no longer tied to XP level** (see *(2026-07-15) physical line becomes a wave-clear pick* entry below; briefly was level-gated at fixed L3/5/8/10 milestones, and even more briefly redesigned into 5 simultaneously-firing skills and reverted the same day, see *(2026-07-13) corrected* entry below); (2) an elemental line ‚Äî the player can invest wave-clear tier upgrades into all 3 elements (Fire/Frost/Lightning) simultaneously, but only one is ever *active* (auto-firing at the nearest enemy, guaranteeing its status effect on hit) at a time; the HUD shows one row per unlocked element (dimmed when inactive), and tapping any row's icon makes that exact element active directly ‚Äî no cycling (see *(2026-07-15) explicit elemental selection* entry below, replacing the earlier single-row cycle-button design from *(2026-07-14) only 1 elemental skill auto-fires*). General stat upgrades (damage/cooldown/+1 arrow/crit/HP/shield/XP-gain, basic-line only) still offered as a 3-choice popup on every XP level-up; elemental tier/damage/cooldown *and* physical-line tier upgrades are all offered together via the wave-clear picker, competing for the same 3 card slots (see dated entries below). Basic Shot/Multishot/Explosive Volley (and any other multi-arrow skill) fire as a fixed angular spread around the nearest enemy's direction, not each arrow independently retargeting its own enemy ‚Äî see *(2026-07-15) fixed-spread multishot* entry below.

## 2. Skills & Lessons

Same base playbook as TopDownAdventure: the **`godot-2d-game-dev`** skill (`C:\Users\nhanp\.claude\skills\godot-2d-game-dev\SKILL.md`) covers the general techniques (verify loop, sprite extraction pipeline, Plan Mode conventions, correctness checklist, GH Pages deploy). What's specific to *this* project:

- **Strategy-pattern behaviors over full component-nodes.** `MovementBehavior`/`AttackBehavior` are stateless `Resource` subclasses shared across every `EnemyData` that references them (Godot dedupes `.tres` Resources by path at runtime ‚Äî several species really do share one instance). This was a deliberate choice over splitting `EnemyBase` into full component Nodes (Health/Hitbox/HitFlash as separate nodes), made when comparing against an external architecture template ‚Äî the reasoning: this project already commits to Resource-driven data everywhere, and full component-nodes would be inconsistent with that for no functional gain at this project's scale. **Any new behavior subclass must never store per-instance state on itself** ‚Äî only read/write fields on the `enemy` node argument passed to every method.
- **The `:=` type-inference gotcha bit this project twice.** `var x := some_expr` fails to compile if the expression's static type resolves to `Variant` ‚Äî most commonly when a variable typed as plain `Node` (e.g. from `get_first_node_in_group()`) has a member accessed that only exists on a subtype like `Node2D` (`.global_position`). Fix: use plain `=` instead of `:=` for anything chained off a generically-typed lookup, or explicitly type the intermediate variable (e.g. `Array[Node2D]` return types sidestep this entirely).
- **Don't assume a Godot API exists ‚Äî verify it.** `GameManager` originally connected to `get_tree().current_scene_changed` to reset pause state on scene reload; that signal doesn't exist on `SceneTree` in Godot 4 at all, and it silently would have been a real runtime error the first time a Restart button was pressed. Fixed by adding an explicit `GameManager.reset_state()` method called directly from both restart handlers, right before `get_tree().reload_current_scene()` ‚Äî more robust than a signal hook anyway, since it's deterministic rather than dependent on scene-tree timing.
- **`SceneTree.create_timer()` defaults `process_always` to `true`.** Any gameplay-critical timer (the boss's telegraph/cooldown/summon loop, a trap's arm/expire timers) needs `create_timer(duration, false)` explicitly, or it keeps counting down in real time even while `get_tree().paused = true` ‚Äî the boss could keep landing hits on a "paused" screen otherwise. This is *not* what you want for music/UI-adjacent timers (e.g. the post-boss-victory music-restart delay), which should keep the default.
- **Centralize signal emission at the state-owner, not at every caller.** `SignalBus.enemy_hit` was originally emitted separately from `projectile.gd` and `trap_zone.gd` (each place that deals damage) ‚Äî this silently missed the "instant_bomb" item effect, which damages enemies directly. Moved into `EnemyBase.take_damage()`/`BossBase.take_damage()` themselves so every current and future damage source is covered by construction.
- **No proactive headless test runs** ‚Äî a standing, explicit user preference for this project specifically (headless Godot runs were found to hang/feel stuck). Verify changes via direct code re-reading instead, and say so plainly when a change hasn't been playtested in-editor.
- **Confirm before every `git push`**, every time, even mid-flow.
