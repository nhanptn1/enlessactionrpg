## 1. Summary

**What it is**: a Godot 4.7 vertical-portrait (720x1280) wave-survival auto-battler — a fixed-position archer at the bottom of the screen, enemies fall from the top, auto-fire combat, XP/leveling, boss every 10 waves. Architecture is deliberately **per-Resource data-driven** (`EnemyData`/`SkillData`/`ItemData`/`WaveData`, plus `MovementBehavior`/`AttackBehavior` Strategy-pattern Resources) — this is a real, intentional divergence from the sibling TopDownAdventure project's const-Dictionary pattern; the two are not meant to converge.

## 2. Skills & Lessons

Same base playbook as TopDownAdventure: the **`godot-2d-game-dev`** skill (`C:\Users\nhanp\.claude\skills\godot-2d-game-dev\SKILL.md`) covers the general techniques (verify loop, sprite extraction pipeline, Plan Mode conventions, correctness checklist, GH Pages deploy). What's specific to *this* project:

- **Strategy-pattern behaviors over full component-nodes.** `MovementBehavior`/`AttackBehavior` are stateless `Resource` subclasses shared across every `EnemyData` that references them (Godot dedupes `.tres` Resources by path at runtime — several species really do share one instance). This was a deliberate choice over splitting `EnemyBase` into full component Nodes (Health/Hitbox/HitFlash as separate nodes), made when comparing against an external architecture template — the reasoning: this project already commits to Resource-driven data everywhere, and full component-nodes would be inconsistent with that for no functional gain at this project's scale. **Any new behavior subclass must never store per-instance state on itself** — only read/write fields on the `enemy` node argument passed to every method.
- **The `:=` type-inference gotcha has bitten this project repeatedly (3+ times).** `var x := some_expr` fails to compile if the expression's static type resolves to `Variant` — most commonly when a variable typed as plain `Node` (e.g. from `get_first_node_in_group()`) has a member accessed that only exists on a subtype like `Node2D` (`.global_position`), but a subtler variant hits even a *typed* array/Dictionary: indexing a plain `Dictionary`'s value inside an `Array[Dictionary]` (e.g. `status_rolls[0]["element"]`) still resolves to `Variant`, and one of these actually blocked the game from starting (a real compile error, not just a caught-on-re-read issue) before it was found. Fix: use plain `=` instead of `:=` for anything chained off a generically-typed lookup or Dictionary-of-typed-array indexing, or explicitly type the intermediate variable.
- **Confirm ambiguous spec readings directly before building around them.** A design doc's tree-diagram mention of "SkillInventory" was read as "5 skills firing simultaneously" and built out fully (a new Timer-per-skill architecture, wave-clear picker changes, HUD rework) before a playtest revealed the actual intent was one evolving skill line — a full day's build had to be reverted. The doc was genuinely ambiguous on this specific point; it deserved a direct question up front alongside whatever other design-detail questions were already being asked, not just those.
- **Don't assume a Godot API exists — verify it.** `GameManager` originally connected to `get_tree().current_scene_changed` to reset pause state on scene reload; that signal doesn't exist on `SceneTree` in Godot 4 at all, and it silently would have been a real runtime error the first time a Restart button was pressed. Fixed by adding an explicit `GameManager.reset_state()` method called directly from both restart handlers, right before `get_tree().reload_current_scene()` — more robust than a signal hook anyway, since it's deterministic rather than dependent on scene-tree timing.
- **`SceneTree.create_timer()` defaults `process_always` to `true`.** Any gameplay-critical timer (the boss's telegraph/cooldown/summon loop, a trap's arm/expire timers) needs `create_timer(duration, false)` explicitly, or it keeps counting down in real time even while `get_tree().paused = true` — the boss could keep landing hits on a "paused" screen otherwise. This is *not* what you want for music/UI-adjacent timers (e.g. the post-boss-victory music-restart delay), which should keep the default.
- **Centralize signal emission at the state-owner, not at every caller.** `SignalBus.enemy_hit` was originally emitted separately from `projectile.gd` and `trap_zone.gd` (each place that deals damage) — this silently missed the "instant_bomb" item effect, which damages enemies directly. Moved into `EnemyBase.take_damage()`/`BossBase.take_damage()` themselves so every current and future damage source is covered by construction.
- **Headless Godot runs are now a core, actively-used verification method** — this bullet used to say the opposite ("no proactive headless runs, they hang"), but that guidance is stale: Godot 4.7 is installed locally specifically for this, and throwaway headless test scenes (built, run, deleted) plus a permanent `scripts/test/systems_smoke_test.gd` suite are now the standard way to verify real runtime behavior, not just that code parses. What *did* turn out to cause an apparent "hang": a headless process whose main scene script fails to parse just sits idle forever (since `_ready()` never runs, so `get_tree().quit()` is never reached) — not an inherent Godot problem. Always wrap headless test invocations in a shell `timeout` (e.g. `timeout 60 godot --headless ...`) as a safety net, and if a run seems stuck, check the actual process's CPU usage/parse errors before assuming it's a slow load. In-editor playtesting by the user is still separate and still needed — headless verification confirms code correctness and runtime behavior, not feel/balance/visuals.
- **Confirm before every `git push`**, every time, even mid-flow.
